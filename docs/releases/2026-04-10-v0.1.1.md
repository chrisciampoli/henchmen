# Henchmen v0.1.1 — OSS readiness release

**Date:** 2026-04-10
**Theme:** 7.5/10 → 10/10. Close every deferred TODO, tighten the
supply chain story, and give a stranger a reproducible path from clone
to running stack.

## What changed and why

### 1. Test quarantines: 56 → 0

Phase 7 of the 2026-04-09 expert-panel remediation shipped with 56
integration tests marked `pytest.mark.skip`. That was a compromise — we
had to leave the tests in place until the provider abstraction and
diff-based evaluator changes could be re-plumbed through the fixtures.
The skipped tests were the single most visible eyesore in the repo for
anyone browsing the integration suite.

v0.1.1 un-quarantines every single one:

- `test_forge_pipeline.py` — refactored to inject a mock
  `MessageBroker` and `DocumentStore` via the same pattern the
  `dispatch_client` fixture uses. 18 tests pass.
- `test_end_to_end.py` — stubs `SchemeExecutor.execute` for the happy
  paths and adds `files_changed` to every mocked `OperativeReport` so
  the L8 zero-diff evaluator doesn't override to escalate. 13 tests
  pass.
- `test_mastermind_orchestration.py` — dropped the
  `TestStateMachineIntegration` class (the in-memory state machine was
  deleted in E1), parametrized the scheme-selection tests, and fixed
  the dossier + handler patterns to match the current
  `SchemeExecutor._HANDLERS` registry. 20 tests pass.
- `test_dispatch_pipeline.py::TestDispatchNormalizerIntegration` —
  `publish_task` now accepts an explicit `broker` argument, so tests
  inject a `_MockBroker` directly instead of patching the provider
  registry. 6 tests pass.

Verification: `pytest tests/integration/` returns **144 passed, 0
skipped**.

### 2. Test isolation: `pytest-randomly`

Added `pytest-randomly` to the `[dev]` extras and ran the unit suite
across three random seeds (42, 1234, 9999). No order-dependency bugs
surfaced — the R4 `_isolate_settings` autouse fixture from the expert
review is doing its job.

### 3. `_mock_settings` consolidation

Nine per-file `_mock_settings()` helpers that each hand-rolled a
`MagicMock` have been replaced with calls to the real `Settings`
class via `model_copy(update=...)`. This catches schema drift: when a
field gets added or removed from `Settings`, tests that reference it
break immediately instead of silently passing against a stale mock.

### 4. Supply-chain hardening

All four Dockerfiles now pin base images to real sha256 digests:
- `python:3.12.8-slim-bookworm@sha256:2199a62885a12290dc9c5be3ca0681d367576ab7bf037da120e564723292a2f0`
- `node:20.18-slim@sha256:ffc11dbf16dd0abcbb7b837410601b4d5592db2d03741e13a4a5336ab74d7ccb`

Previously these were placeholder zeros with a `TODO: pin to actual
digest` comment. A vulnerability scan against these exact digests now
has a stable answer.

Firestore security rules are deployed via Terraform. The
`terraform/modules/data-stores` module now wires
`google_firebaserules_ruleset` + `google_firebaserules_release`
resources that deploy `firestore.rules`, interpolating
`var.project_id` into the mastermind SA email regex via
`templatefile`. The `firebaserules.googleapis.com` API is enabled in
`project-bootstrap`. Previously the rules file was a stub with a
comment saying "not deployed by Terraform today".

### 5. Target-repo agnosticism at runtime

The `run_tests` handler inside Mastermind used to be hardcoded for
JS/TS monorepos (pnpm + turbo). A new
`src/henchmen/utils/stack_detector.py` module detects Python, Node
(pnpm/npm), Go, Rust, and Java (Maven/Gradle) by inspecting manifest
files, and the handler routes to the right test / lint commands
based on the detected stack. JS/TS monorepo handling is preserved as a
legacy branch, but the README no longer carries the "hardcoded for
JS/TS" disclaimer.

### 6. BYO-LLM parity — harness improvements, numbers deferred

The eval harness already shipped in the expert-panel remediation. This
release focuses on polish around it rather than running live evals:

- 3 new fixtures: `bugfix_import_error`, `feature_cli_flag`,
  `refactor_extract_function`. Total is now 6.
- `evals/baseline.json` is a structured stub with per-provider
  placeholders and a `how_to_populate` field showing the exact command
  to run on your own hardware. No numbers are hand-written — every
  entry is null until someone runs the harness and opens a PR.
- `.github/workflows/evals.yml` is a new `workflow_dispatch` workflow
  that triggers a single-provider eval run with secrets injected and
  opens a PR with the updated baseline for review.
- `OllamaProvider` now issues a tool-calling capability probe on the
  first `generate()` call that passes any tools. If the selected
  model returns no `tool_calls` on a trivial canary, the provider
  raises a clear error naming the model instead of silently falling
  back to text-only output an hour into a real operative run. Gated
  behind `HENCHMEN_LLM_OLLAMA_SKIP_PROBE` for test doubles.

Live eval numbers are explicitly out of scope for v0.1.1 — the
maintainer doesn't want to embed the cost of a full multi-provider run
in a soft-launch release. The infrastructure is in place; community
contributors can populate the baseline on their own accounts.

### 7. Reliability primitives

Two cross-provider primitives landed on the `DocumentStore` protocol:

- `increment(collection, doc_id, field_deltas)` — atomic counter
  increments. GCP uses `firestore.Increment` transforms. Local SQLite
  uses a per-doc `asyncio.Lock` + read-modify-write. AWS DynamoDB uses
  `UpdateExpression="ADD ..."` on top-level attributes. `TaskTracker`
  now uses this instead of the old read-modify-write block; the
  `TODO(K4-cross-process)` markers are gone.
- `update_if(collection, doc_id, expected_field, expected_value,
  new_values)` — compare-and-set. GCP uses Firestore transactions.
  SQLite uses the per-doc lock + conditional update. DynamoDB uses
  `ConditionExpression` and catches `ConditionalCheckFailedException`.
  `MergeQueue.dequeue` now uses this as an atomic claim; the
  "best-effort FIFO" caveats and all `TODO(E7-transaction)` markers
  are gone.

AWS DLQ pulling is fully implemented. `SNSMessageBroker.pull_dlq`
used to raise `NotImplementedError`; it now uses a lazy boto3 SQS
client, resolves the queue URL via `get_queue_url`, pulls messages
with `receive_message`, decodes bodies to dicts matching the GCP
shape, and acks via `delete_message_batch`. The error path (queue
doesn't exist) raises a clear `RuntimeError` identifying the queue
name.

### 8. Self-hoster experience

- New `docs/deploy-gcp.md` — a linear 30-minute walkthrough from blank
  GCP account to a live stack. Replaces the previous scattering of
  setup hints across `operations.md` and `bootstrap-gcp.sh`.
- `README.md` provider matrix marks AWS as **experimental** and points
  to GitHub Discussions for community-led walkthroughs. GCP and Local
  are the only two **supported** providers for v0.1.1.
- New `henchmen doctor` CLI command runs a local self-check: Python
  version, Docker daemon reachability, git identity, `.env.local`
  discoverability, LLM credentials (provider-aware), operative Docker
  image. Exits non-zero on any failure.
- `docs/operations.md` and `docs/rollback-procedures.md` were rewritten
  in self-hoster voice — no more internal Slack channel references,
  no more assumed-prior-context language. Both link to
  `deploy-gcp.md` instead of duplicating setup instructions.
- `.github/workflows/ci.yml` now has a `docker-compose-smoke` job that
  runs `docker compose up -d`, waits for all services to report
  healthy, and tears down. Catches regressions to healthchecks,
  compose wiring, and base image pins.

### 9. Target-repo scrubbing

The public repo is now completely clean of references to the
maintainer's private target repo. Fixture data uses `acme-org/sample-repo`.
The SECURITY.md contact email is intentional and preserved.

## What's explicitly deferred

- **Live eval numbers.** The harness + workflow ship; the numbers
  don't. Contributors populate via `workflow_dispatch`.
- **AWS walkthrough doc.** AWS is downgraded to experimental. A full
  `deploy-aws.md` happens when someone with AWS budget volunteers to
  test it end-to-end.
- **Hosted demo.** Out of scope for a soft launch.
- **Public metrics dashboard.** The observability stack produces
  Prometheus output and a sample is checked in at
  `docs/images/metrics-sample.txt`, but a live dashboard hosted
  somewhere strangers can see is a follow-up.
- **SLSA Level 3 provenance.** The release workflow is SLSA L1 today
  (SHA-pinned actions, reproducible builds). L3 requires a hosted
  build environment we don't run.

## How to verify these claims yourself

```bash
git clone https://github.com/chrisciampoli/henchmen.git
cd henchmen
pip install -e ".[local,dev]"

# Full test suite — 887 tests, 0 skipped
pytest tests/

# Self-check
henchmen doctor

# Run a single eval fixture against a mock provider (no API keys required
# — the harness substitutes a deterministic in-process LLM double)
henchmen eval --provider local --fixture bugfix_off_by_one

# Terraform static validation against the dev env
cd terraform/environments/dev
terraform init -backend=false
terraform validate
```

None of these commands cost money or require cloud credentials. They
exercise the entire stack except for the real LLM / Pub/Sub / Cloud Run
boundary, which the harness stubs.
